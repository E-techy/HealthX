# 1. System Overview

The Delegated Access System is a secure, granular data-sharing module within the HealthX application.

It allows users to share specific scopes of their health data (e.g., Nutrition Logs, Health Goals) via dynamically generated QR codes.

Instead of exposing permanent access, the system generates secure, revocable **"hashes"** that act as temporary or strictly scoped permission tokens.

The client-side implementation features a cinematic, dark-themed UI with a robust drawer-based navigation system to manage the entire lifecycle of these connections.

---

# 2. Architectural Pattern (MVVM)

The module strictly adheres to the **Model-View-ViewModel (MVVM)** architecture, utilizing Kotlin Coroutines and StateFlow for reactive UI updates.

## Model (Data Layer)

Defines standard API request/response payloads (`ShareableHash`, `BlocklistedUser`, `StandardResponse`).

Uses Retrofit for network calls.

---

## ViewModel (`DelegatedAccessViewModel`)

Acts as the single source of truth for the UI.

It encapsulates all network logic, handles token authorization, and maps raw API responses into a unified `UiState` sealed class (`Idle`, `Loading`, `Success`, `Error`).

---

## View (UI Layer)

100% Jetpack Compose.

Reacts continuously to the `StateFlow` emitted by the ViewModel.

---

# 3. UI Components & Screen Flow

The system is encapsulated within `DelegatedAccessActivity` and navigated via a custom `ModalNavigationDrawer`.

The drawer utilizes an enum-based state manager (`CurrentScreen`) to swap out the main content area without launching new activities.

---

## A. The Access Gateway Drawer

The central navigation hub for the module, featuring three primary destinations:

- **My QR Code** — The generator screen.
- **Active Shared Links** — The management screen for active hashes.
- **Blocked Users** — The management screen for revoked/blocked connections.

---

## B. Screen 1: QR Generator & Permissions (`QRCodeGeneratorScreen`)

### Default State

Displays a static QR code containing the user's base `userId` with the category `ADD_FRIEND`.

### Permissions Bottom Sheet

Triggered via the **"Create Custom QR Code"** button.

Iterates through `AccessPermissions.availablePermissions` to render granular toggle switches.

Examples include:

- View Nutrition
- Set Goals

### Custom Hash State

Upon successful API generation, the UI switches out the base QR code for the newly generated secure Hash QR (`SHARE_ACCESS` category).

The generated QR renders in a green success state and includes a quick-reset button to revert to the default QR.

---

## C. Screen 2: Active Links Manager (`ActiveLinksScreen`)

### List View

Fetches and displays all active hashes generated by the user.

### Card Details

Displays:

- Truncated Hash ID
- List of permissions attached to that hash

### Interactive Controls

- **Toggle Switch**
    - Instantly pauses/resumes the hash via a `PATCH` request.

- **Delete Icon**
    - Permanently revokes and destroys the hash via a `DELETE` request.

---

## D. Screen 3: Blocklist Manager (`BlocklistScreen`)

### List View

Displays users who have been actively blocked from scanning hashes.

### Visuals

Renders the blocked user's profile image using **Coil AsyncImage**.

If unavailable, a generated initials avatar is displayed.

### Controls

Contains an **Unblock** action that executes a `DELETE` request to the blocklist endpoint and immediately refreshes the local UI state upon success.

# 4. API Integration & State Management

All network requests require a valid JWT passed in the following header:

```http
Authorization: Bearer <token>
```

## API Endpoints

| Feature | HTTP Method | Endpoint | Action |
|----------|------------|----------|--------|
| Generate Hash | POST | `/api/access/hash` | Takes selected boolean actions and returns a secure 64-character hex string. |
| Get Hashes | GET | `/api/access/hash` | Retrieves all `ShareableHash` objects. |
| Toggle Hash | PATCH | `/api/access/hash/:hashId/status` | Updates status between `ACTIVE` and `UNACTIVE`. |
| Delete Hash | DELETE | `/api/access/hash/:hashId` | Permanently destroys the hash document. |
| Get Blocklist | GET | `/api/access/blocklist` | Retrieves joined user data for blocked accounts. |
| Unblock User | DELETE | `/api/access/blocklist/:userId` | Removes the user from the blocklist database. |

---

## State Handling (`UiState` Sealed Class)

To prevent UI crashes and handle asynchronous data cleanly, every API call emits states to a `MutableStateFlow`.

```kotlin
sealed class UiState<out T> {
    object Idle : UiState<Nothing>()
    object Loading : UiState<Nothing>()
    data class Success<T>(val data: T) : UiState<T>()
    data class Error(val message: String) : UiState<Nothing>()
}
```

# 5. Security & Error Handling

## Thread Safety

The QR code generation (`QRGenerator.generateQRCode`) operates on `Dispatchers.Default` to prevent heavy matrix calculations from blocking the main UI thread.

---

## Standardized Error Parsing

The backend emits a standardized JSON error model:

```json
{
  "success": false,
  "message": "..."
}
```

The ViewModel intercepts `4xx` and `5xx` responses, uses Gson to parse the `errorBody()`, and extracts the human-readable message.

---

## UI Feedback

Errors are emitted through the `UiState.Error` state.

A `LaunchedEffect` in the Compose UI observes this state and displays a non-intrusive `Snackbar`, informing the user of the error without crashing the application.
